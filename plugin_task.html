Task management

In Ikomia platform, a task is the implementation of a specific algorithm. We use inheritance to set a functional framework and simplify customization. Basically, any new Ikomia plugin must implement a task class derived from the base class CWorkflowTask or from a specialization class which itself inherits from this base class.

  • Recommended methods

  • Metadata

Mandatory methods to implement

  • Constructor: base class contructor must be called explicitly. This is also the place where you define inputs and outputs.

class MyAlgorithm(dataprocess.CWorkflowTask):

    def __init__(self, name, param):
        dataprocess.CWorkflowTask.__init__(self, name)
        # Add image input
        self.addOutput(dataprocess.CImageIO())
        # Add graphics output
        self.addOutput(dataprocess.CGraphicsOutput())
        # Add numeric output
        self.addOutput(dataprocess.CNumericIO())

  • run(): launch the algorithm execution. When the method is called, inputs data are set and ready to process. The method must begin with a call to beginTaskRun() and end with a call to endTaskRun(). In the code snippet below, you will find the most useful methods that you will need while implementing your own plugin:

class MyAlgorithm(dataprocess.CWorkflowTask):

    def run(self):

        # Execution initialization
        self.beginTaskRun()

        # Get inputs
        input1 = self.getInput(0)
        input2 = self.getInput(1)

        # Get parameters
        param = self.getParam()

        # Get outputs
        output1 = self.getOutput(0)
        output1 = self.getOutput(1)

        # Execution finalization
        self.endTaskRun()

Note

See CWorkflowTask to have details around all available methods.

Recommended methods to implement

  • getProgressSteps(): Ikomia Studio only. For time consuming algorithms, it could be relevant to split execution into separate parts and give progression feedback after each part (if possible). If so, you have to first implement this method and return the number of execution parts. Then, in your run() method, you have to place a call to self.emitStepProgress() after each part to notify the main progress bar of Ikomia Studio.

  • stop(): method called when a user requests the process to stop. For time consuming algorithm it’s highly recommended to integrate a stop mechanism inside your algorithm execution code. This method is somehow just a callback.

Available specializations

To ease implementation of common image processing tasks, we provide some specialization classes from which you can inherit. They integrate useful features to speed your developpment.

Simple image processing algorithms

C2dImageTask : base class dedicated to algorithms processing image and producing at least a result image.

  • Inputs: image (CImageIO) and graphics (CGraphicsInput).

  • Outputs: image (CImageIO).

  • Important features:

    • create binary mask from graphics. See createGraphicsMask().

    • apply a binary mask to the desired output image so that only masked areas are processed. See applyGraphicsMask() and applyGraphicsMaskToBinary().

    • forward input image to the desired output without modification: forwardInputImage().

    • attach a color overlay mask to the desired output (Ikomia Studio only). See setOutputColorMap().

Note

If any of these features are useful for your own algorithm, inherit your class from this base class instead of CWorkflowTask. Please consult this tutorial for details.

Interactive image processing algorithms

C2dImageInteractiveTask: choose it as a base class if you need user interaction for your algorithm. The class embeds a graphics interaction layer. Every changes made by users on this layer are notified and corresponding actions can be implemented. Ikomia Studio only.

  • Inputs: image (CImageIO) and graphics (CGraphicsInput).

  • Outputs: image (CImageIO).

  • Important features:

    • callback to manage interaction layer modification. See graphcisChanged().

    • create binary mask from the graphics interaction layer. See createInteractionMask().

    • compute connected components from the binary mask generated from the interaction layer. See computeBlobs().

    • get connected components as a list of polygons (list of points). See getBlobs().

    • clear interaction layer. See clearInteractionLayer().

Note

If any of these features are useful for your own algorithm, inherit your class from this base class instead of CWorkflowTask.

Video processing algorithms

CVideoTask: choose it as a base class if your algorithm is dedicated to video or stream. The class inherits from C2dImageTask adding only a mechanism to handle start and stop events.

  • Inputs: image (CImageIO) and graphics (CGraphicsInput).

  • Outputs: image (CImageIO).

  • Important features:

    • callback to manage video start event. See notifyVideoStart().

    • callback to manage video stop event. See notifyVideoStop().

Note

Please consult source code of infer_raft_optical_flow for implementation example.

Deep Learning training algorithms

TrainProcess: choose it as a base class for any deep learning training algorithms. This base class integrated training monitoring features for MLflow and Tensorboard.

  • Inputs: Ikomia dataset (IkDatasetIO).

  • Outputs: None

  • Important features:

    • MLflow: a local tracking server is automatically started and ready to use at http://localhost:5000. By default, logging data are saved in the Ikomia folder in mlflow subdirectory. The class offers a thin wrapper to access logging methods: model hyper-parameters with log_param() and log_params(), model metrics with log_metric() and log_metrics() and model artifacts with log_artifact() and log_artifacts().

    • Tensorboard: a local tracking server is automatically started and ready to use at http://localhost:6006. By default, logging data are saved in the Ikomia folder in tensorboard subdirectory. Use classic Tensorboard functions to log your data.

Note

Please consult source code of train_yolov5 for implementation example.

Metadata

Ikomia platform allows you to add additionnal information to your plugin. This will help users to know more about your algorithm and give essential visual elements in Ikomia HUB.

These information are store in a factory class inherited from CTaskFactory. It aims to create instance of your process class. Plugin metadata are set in the constructor within the CTaskInfo member variable. You will find he full list of information in the CTaskInfo documentation.

For Ikomia Studio, you can customize the documentation page of your plugin. It can be interesting if you want to put more information than the provided fields. You just need to place your documentation page in the root folder of the plugin. File format can be either Markdown or HTML. Ikomia Studio will automatically search for files with the following names: doc.md, doc.html, doc.htm, documentation.md, documentation.html, documentation.htm, info.md, info.html, info.htm.
